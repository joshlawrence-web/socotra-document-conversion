# Pipeline Test Suite

End-to-end integration tests for the five-leg document pipeline.
Covers the full path: DOCX → Leg 0 → conditional form → Leg 2+3 → Leg 4 (plugin).

## Quick start

```bash
# Step 1: generate the DOCX fixture files (one-time, or after changing fixtures)
python3 scripts/generate_test_fixtures.py

# Step 2: run the full pipeline in automated mode
python3 tests/pipeline/run_test_pipeline.py --auto
```

Output lands in `tests/pipeline/output/<stem>/`.

---

## Modes

| Flag | Behaviour |
|------|-----------|
| *(none)* | Interactive — pauses after each Leg 0 run so you can hand-fill the conditional form |
| `--auto` | Automated — fills conditions from `condition_seeds.yaml` (CI-safe, no pauses) |
| `--only "TestItemCert(segment)"` | Run just one fixture (stem match, partial match works) |
| `--regen` | Regenerate DOCX fixtures before running |
| `--no-leg4` | Skip Leg 4 plugin generation |

---

## Test fixtures

Stored in `tests/pipeline/fixtures/`. Generated by `scripts/generate_test_fixtures.py`.

| File | Type | Fields | Conditionals |
|------|------|--------|--------------|
| `TestQuoteSummary(quote).docx` | quote | quoteNumber, account, startTime, endTime, jurisdiction, quotePremiumTotal, quoteOtherTotal | 2 |
| `TestItemCert(segment).docx` | segment | policyNumber, account, item (type/date/price/serial), coverage | 3 |
| `TestRenewalNotice(segment).docx` | segment | policyNumber, account, contractTermEndDate, expectedRenewalDate, discount, termChargesTotal | 3 |

Each document uses real registry accessor paths in `{field}` tokens so Leg 2 produces
high-confidence matches. Conditional blocks use `[[text]]` syntax so Leg 0 generates
a valid conditional form.

---

## Condition seeds

`condition_seeds.yaml` maps each document stem to a dict of `block_id → condition_expression`.
Conditions use valid accessor paths from `registry/path-registry.yaml`.

To add a new fixture:
1. Add a builder function in `scripts/generate_test_fixtures.py` and append it to `FIXTURES`.
2. Add the fixture filename to `ALL_FIXTURES` in `run_test_pipeline.py`.
3. Add condition seeds for its blocks in `condition_seeds.yaml`.
4. Run `python3 scripts/generate_test_fixtures.py` to write the new DOCX.

---

## What is tested

Each fixture run validates:
- **Leg 0** exits 0 and writes `.raw.html`, `.annotated.html`, `.mapping.yaml`, `.conditional-form.md`
- **Conditional form parse** exits 0 and writes `.conditional-registry.yaml`
- **Leg 2+3** exits 0 and writes `.final.vm`

The combined **Leg 4** run validates:
- All successful mappings can be combined into a single `ZenCoverDocumentDataSnapshotPluginImpl.java`
- Additive mode works when more than one mapping is processed

## Running in CI

```bash
# CI command (no interactive pauses, non-zero exit on failure)
python3 tests/pipeline/run_test_pipeline.py --auto
```

Add `--regen` if fixtures are not committed to the repo.

---

## Troubleshooting

**"fixture file(s) missing"** → Run `python3 scripts/generate_test_fixtures.py` first.

**"Leg 0 failed"** → Check that `python-docx` is installed: `pip install python-docx --break-system-packages`

**"Leg 2+3 failed"** → Check `tests/pipeline/output/<stem>/<stem>.review.md` for unresolved tokens.

**Conditions not matching** → Update `condition_seeds.yaml` with valid accessor paths.
Run `python3 scripts/list_paths.py` to see the full registry.
