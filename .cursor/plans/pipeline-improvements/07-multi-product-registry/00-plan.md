## 1. Background

The converter is currently hardcoded around a single product config (`socotra-config/` → ZenCover). All registry files live flat in `registry/`. A second product would require either:

(a) Overwriting `registry/path-registry.yaml` (destructive, breaks ZenCover runs)  
(b) Manually copying the whole `registry/` folder and updating every `registry=` arg in every command

Neither is documented. This plan adds:
1. A convention for per-product registry directories: `registry/<ProductName>/`
2. A scaffold script that generates a skeleton `path-registry.yaml` from a Socotra config directory
3. CLAUDE.md + README guidance on the multi-product workflow
4. `agent.py` auto-detection: if `registry=` is omitted and multiple product registries exist, prompt the user to choose (or accept `product=` kwarg)

## 2. Decisions

| # | Topic | Decision |
|---|-------|----------|
| D1 | Directory convention | `registry/<ProductName>/path-registry.yaml` (and co-located `sdk-schema-index.yaml`, `terminology.yaml`, `skill-lessons.yaml`). The flat `registry/*.yaml` files remain as the default (no migration required for ZenCover). |
| D2 | Default resolution | When `registry=` is omitted, look for `registry/path-registry.yaml` first (existing behaviour). If that exists, use it. If it doesn't, look for `registry/*/path-registry.yaml` and pick if exactly one. If ambiguous, print choices and exit 1. |
| D3 | `product=` kwarg | Add `product=<Name>` to `RUN_PIPELINE` as an alias for `registry=registry/<Name>/path-registry.yaml`. Convenience only — `registry=` explicit path still works. |
| D4 | Scaffold script | `scripts/scaffold_registry.py` — reads a Socotra config directory and generates a skeleton `path-registry.yaml` with correct `system_paths` (always the same) and stub `exposures` entries derived from the config's product YAML. Does not require JARs. |
| D5 | Scaffold output | Writes to `registry/<ProductName>/path-registry.yaml` by default. Prompts before overwriting. |
| D6 | sdk-schema-index per product | Each product gets its own `sdk-schema-index.yaml` (JARs differ). The scaffold script notes this but does not run `build_schema_index.py` — that requires JARs. |
| D7 | CLAUDE.md | Add a "Multi-product setup" section with the scaffold command and the `product=` kwarg examples. |
| D8 | No breaking changes | Existing ZenCover workflows unchanged. The flat `registry/` layout is still valid indefinitely. |

## 3. Task list

### T1 — `scripts/scaffold_registry.py`

**Goal:** Generate a skeleton `path-registry.yaml` for a new product.

**Algorithm:**
1. Read the Socotra config directory (`--config-dir`)
2. Find `product.yaml` (or equivalent) — extract product name, exposures list
3. For each exposure, create a stub entry in `exposures[]` with `name`, empty `fields`, empty `coverages`
4. Copy `system_paths` from a built-in template (these are always the same across Socotra products)
5. Add placeholder `account_paths`, `policy_data`, `policy_charges`, `datafetcher_paths` with comments instructing the user to fill them

**CLI:**
```bash
python3 scripts/scaffold_registry.py \
  --config-dir socotra-config/ \
  --product ZenCover \
  --out registry/ZenCover/path-registry.yaml
```

**Output:** A valid (but mostly-stub) `path-registry.yaml` with:
- Correct `schema_version` and `product` header
- All `system_paths` populated (locator, policyNumber, policyStartTime, etc.)
- Exposure stubs from config
- `# TODO: fill in fields for this exposure` comments
- A top comment: `# Generated by scaffold_registry.py — fill in fields and coverages before running Leg 2`

**Files:** `scripts/scaffold_registry.py` (new)

---

### T2 — `product=` kwarg in `agent.py`

**Goal:** Accept `product=<Name>` as an alias for `registry=registry/<Name>/path-registry.yaml`.

Resolution order in `agent.py`:
1. If `registry=` explicitly given → use it
2. Elif `product=` given → `registry=registry/<product>/path-registry.yaml`
3. Elif `registry/path-registry.yaml` exists → use it (existing default)
4. Elif exactly one `registry/*/path-registry.yaml` exists → use it (with a logged note)
5. Else → print available products and exit 1

**Files:** `scripts/agent.py`
